# GenOS Knowledge Capture

GenOS 현장 작업(FDE)에서 *한 번 배운 걸 다음 세션·다음 고객사에서 다시 파헤치지 않도록* 만드는 개인 스킬이다. AI의 세션 기억은 끊기지만, vault에 원자적 노트로 남기고 **MOC(허브)에서 엮어두면** 새 세션이 그 한 장만 읽고 빠르게 온보딩한다.

이 스킬은 사용자 개인 자산(`~/.claude/skills/`)이다. **회사 레포의 `.claude/skills/genos-*`(genos-session-close 등)는 회사 공유 자산이라 절대 건드리지 않는다.** 세션 상태는 회사 HANDOFF가, 재사용 지식은 이 스킬이 담당한다 — 둘은 분리된 관심사다.

## 핵심 원칙 (왜 이렇게 하나)

온보딩 가속의 실체는 "그래프뷰가 예쁜 것"이 아니라 **작은 진입점(MOC)을 먼저 읽고, 거기 링크 중 오늘 필요한 1~2개만 선택적으로 읽는 것**이다. AI는 Obsidian 그래프를 순회하지 못한다. 그래서 이 스킬은 늘 두 가지를 함께 한다: ① 원자적 노트 작성, ② **MOC에 링크 추가**. MOC에 안 걸린 노트는 새 세션이 발견하지 못하는 고아가 된다.

토큰 절감은 "재탐색(re-grep·재독)을 대체할 때 + 선택적으로 1~2개만 읽을 때"만 생긴다. 그러니 노트는 **재사용 가능한 일반화(패턴·함정·결정)**만 담아야 한다. 오늘 무슨 파일을 고쳤는지 같은 휘발성 상태를 vault에 넣으면 노이즈만 쌓이고 검색 가치가 떨어진다.

가독성의 기본값은 **story-first field note**다. GenOS 지식 노트는 새 세션의 AI뿐 아니라 사용자가 고객사 현장에서 설명할 수 있어야 한다. 먼저 "고객이 어떤 문제를 갖고 있고, GenOS가 어디서 가치를 만드는가"를 이야기로 잡고, 그 다음 bullet/table/Mermaid로 판단과 근거를 스캔 가능하게 만든다. 긴 소스 경로는 본문 곳곳에 흩뿌리지 말고 `근거 지도`로 모은다.

중요: **story-first는 장문 에세이가 아니다.** 첫머리는 이야기로 열되, 판단·분기·함정·절차는 bullet, 표, Mermaid로 즉시 읽히게 쓴다. bullet은 여전히 중요한 표현 방식이지만 노트의 spine은 "고객 설명 → FDE 판단 → 기술 근거 → 현장 체크" 순서다. 내용을 줄여야 할 때는 "세션 상태/중복/일회성 정보"만 제거하고, 판단 근거·예외·trade-off는 보존한다.

## 3계층 — 무엇을 캡처하고 무엇을 버리나

캡처 후보가 생기면 먼저 어느 계층인지 판별한다. 계층을 섞지 않는 것이 vault가 비대해져도 길을 잃지 않는 유일한 규칙이다.

| 계층 | 무엇 | 저장 위치 | 캡처? |
|------|------|----------|-------|
| **플랫폼** | GenOS 자체 구조·패턴·함정 (모든 고객사 재사용) | `Projects/GenonAI/GenOS/` | ✅ |
| **고객사** | 특정 고객 커스텀 결정·맥락 (삼성카드 등) | `Projects/GenonAI/<고객사>/` 또는 `GenonAI/` loose | ✅ |
| **세션 상태** | 오늘 진행/대기/다음 작업 | vault 밖 — GenOS 레포 `HANDOFF_NOW.md` | ❌ |

판단 기준: **"이걸 6개월 뒤 다른 세션·다른 고객사에서 다시 보면 시간을 아껴줄까?"** Yes면 플랫폼/고객사 노트, No(이번 작업에만 유효)면 캡처하지 않는다.

Vault 루트: `~/Library/Mobile Documents/iCloud~md~obsidian/Documents/Note`
GenOS 영역: `<vault>/Projects/GenonAI/`

## 워크플로우

순서엔 실제 의존성이 있다(중복 확인 → 작성 → MOC 갱신). 하지만 **비협상은 둘뿐이다: ② 작성 전 중복 확인, ⑤ MOC 갱신.** 나머지는 상황에 맞게 조정해도 된다 — 후보가 하나면 계획 제시를 짧게, 명백하면 확인을 생략하는 식으로.

### 1. 후보 추출 — 이번 세션에서 일반화 가능한 것만
대화·작업 내역에서 재사용 지식을 골라낸다. 좋은 후보: 비자명한 패턴, 빠지기 쉬운 함정, 협의로 확정된 설계 결정, "문서엔 X라는데 실제론 Y"인 사실. 나쁜 후보: 진행상황, 파일 목록, 커밋 해시, 이번 한정 상태.

### 2. 계층 판별 + 중복 확인 (중요)
각 후보가 플랫폼/고객사 중 어디인지 정한다. 그다음 **기존 노트를 먼저 검색**한다:
```bash
ls "<vault>/Projects/GenonAI/GenOS/" "<vault>/Projects/GenonAI/"
grep -rl "<핵심 키워드>" "<vault>/Projects/GenonAI/"
```
같은 주제 노트가 있으면 **새로 만들지 말고 그 노트를 갱신**한다(원자성 유지, 중복 금지).

### 3. 사용자에게 캡처 계획 제시 → 확인
글을 쓰기 전, 무엇을 어디에 쓸지 짧게 보여주고 확인받는다. 사용자는 위임 엔지니어 방식을 선호하므로 길게 묻지 말고 목록만:
```
[캡처 계획]
- GenOS/<제목>.md (플랫폼·신규)  — <한 줄 요지>
- <고객사>/<제목>.md (고객사·기존 갱신) — <무엇 추가>
- MOC: "패턴·함정" 섹션에 링크 추가
```
확인 후 작성. 사용자가 "그냥 해" 류면 확인 생략 가능.

### 4. 노트 작성 — Zettelkasten 형식 (vault 기존 컨벤션 준수)
`obsidian-note` 스킬과 동일 형식. **단, Tavily 웹 리서치는 생략한다** — 출처는 웹 개념이 아니라 이 세션의 실작업·레포다.

Frontmatter:
```yaml
---
source:
  - "admin-api/src/main.py"               # repo-relative evidence identifier
  - "origin/project/samsung-card/feat/96:admin-api/src/util/pims_utils.py"
  - "SAMSUNGCARD-96"
related_notes:
  - "[[실존하는_노트만]]"                  # 반드시 vault에 존재하는 노트
tags:
  - work/genos
  - type/pattern        # 또는 type/moc, type/onboarding
  - customer/samsung-card  # 고객사 노트일 때
  - feature/<기능>         # 횡단 검색용
created: YYYY-MM-DD        # 오늘 날짜
---
```
`source`는 Obsidian 클릭 링크가 아니라 근거 식별자로 쓴다. URI-style prefix나 개인 PC 절대경로를 붙이지 않는다. GenOS 레포 파일은 `admin-front/src/layouts/UserLayout.js`처럼 repo-relative path만 쓴다. 특정 브랜치 파일은 `origin/project/samsung-card/feat/96:admin-api/src/util/pims_utils.py`처럼 `git show <branch>:<path>`에 바로 넣을 수 있는 형태로 쓴다. 다른 repo가 근거면 `genos-chart/helmfile.yaml`처럼 repo 이름을 첫 path segment로 둔다. 이슈는 `SAMSUNGCARD-96`처럼 사람이 바로 읽는 식별자로 쓴다. 본문 근거는 필요하면 `admin-front/src/layouts/UserLayout.js:95`처럼 repo-relative `file:line`으로 인용한다.

본문 기본형은 **story-first field note**다:

- **`## 한 문장 결론`** — 고객사 현장에서 바로 말할 수 있는 결론 1문장.
- **`## 고객에게 90초로 설명하면`** — 기능 정의보다 고객 문제와 GenOS의 역할을 먼저 설명한다.
- **`## 왜 필요한가`** — 이 기능이 없으면 현장에서 무엇이 막히는지, 어떤 운영/보안/분석 문제가 생기는지 쓴다.
- **`## FDE가 먼저 잡아야 하는 큰 흐름`** — 고객 질문을 어떤 축으로 나눠 판단할지 bullet로 정리한다.
- **`## 전체 흐름`** — 첫 Mermaid 또는 ASCII diagram. 전체 책임 경계와 주요 경로를 한눈에 보여준다.
- **`## 고객 질문에 이렇게 답한다`** — 실제 고객 질문을 heading으로 두고 답변 문장 + 조건/주의를 붙인다.
- **`## 실제 GenOS에서는 어떻게 움직이나`** — 런타임 흐름, API 흐름, 배치 흐름을 필요한 만큼 diagram과 bullet로 쓴다.
- **`## 현장 리스크와 확인 포인트`** — 보안, 네트워크, 권한, 장애, 운영 경계처럼 고객사에서 먼저 확인할 내용을 모은다.
- **`## 전문적으로 설명해야 할 때`** — 엔지니어/아키텍트에게 말할 내부 구조와 trade-off를 쓴다.
- **`## 엔지니어 딥다이브`** — 코드 레벨 구조, enum, schema, DB model, helper pattern을 보존한다.
- **`## 데모/시연 스크립트`** — 고객 앞에서 어떤 순서로 보여주고 어떤 관찰 포인트를 말할지 쓴다.
- **`## 현장 체크리스트`** — 실제 작업 전 확인할 항목을 체크박스로 남긴다.
- **`## 근거 지도`** — 소스 경로와 line reference는 가능하면 여기로 모은다.
- **`## 다음에 확인할 것`** — 구체적 후속 질문. 키워드 나열 금지.

모든 노트가 모든 섹션을 강제로 가져야 하는 것은 아니다. 고객 설명/운영 판단/기술 근거가 모두 필요한 플랫폼 노트는 위 spine을 따른다. 짧은 고객사 결정 노트나 단일 함정 노트는 핵심 섹션만 남기되, "고객에게 어떻게 말할지"와 "근거가 어디인지"는 빠뜨리지 않는다.

Mermaid 사용 원칙:

- Mermaid는 **1개로 제한하지 않는다**. 각 diagram은 하나의 질문에 답해야 한다.
- 텍스트 100마디보다 도표 하나가 빠르면 적극 사용한다. 장식용 diagram은 넣지 않는다.
- 첫 diagram은 전체 맥락을 잡는 용도여야 한다. 이후 diagram은 runtime sequence, branching, lifecycle, data relationship처럼 섹션별 이해 병목을 풀 때 추가한다.
- `flowchart`는 책임 경계/분기, `sequenceDiagram`은 요청 순서, `stateDiagram`은 상태 변화, `erDiagram`은 데이터 관계, `graph`는 시스템 구성요소 관계에 쓴다.
- Mermaid가 있더라도 바로 아래에 "이 그림에서 봐야 할 점" bullet 2~4개를 붙인다.

문체 규칙:

- 정보량은 보존한다. story-first로 바꾸면서 내용, 예외, 판단 근거, trade-off를 삭제하지 않는다.
- 첫머리는 사람이 읽는 문장으로 열고, 이후 판단은 bullet/table/diagram으로 스캔 가능하게 만든다.
- 한 문단은 4문장 이하. 5문장 이상이면 bullet 또는 diagram으로 쪼갠다.
- bullet은 "소제목 → 설명 → 근거 → 주의"처럼 여러 단계로 깊어져도 된다. 단, 각 줄은 한 판단만 담는다.
- `file:line` 근거가 3개 이상 필요하면 본문에 흩뿌리지 말고 `근거 지도`로 보낸다.
- 표는 "분류/판단"에만 쓴다. 긴 설명을 표 안에 우겨 넣지 않는다.
- 소스 경로가 문장의 주어가 되면 가독성이 떨어진다. 먼저 판단을 쓰고, 경로는 마지막에 붙인다.

### 초심자 친화 캡처 패턴

GenOS 지식 노트는 새 세션의 AI뿐 아니라 **입사한 지 얼마 안 된 사람이 처음 보는 프로젝트를 따라갈 때**도 읽을 수 있어야 한다. 코드 위치와 근거만 촘촘히 나열하면 AI에게는 편하지만 사람에게는 막힌다. 특히 플랫폼 구조·권한·배포·운영처럼 처음 보는 사람이 용어에서 멈출 수 있는 주제는 아래 패턴을 우선 적용한다.

필요하면 아래 섹션을 추가한다:

- **`## 읽는 모드`** — 고객 설명 90초, FDE 판단 5분, 엔지니어 딥다이브 20분처럼 독자의 읽는 깊이를 안내한다.
- **`## 이 노트에서 쓰는 말`** — 짧은 표. 용어 설명은 한 문장 이하.
- **`## 전체 흐름`** — Mermaid를 우선 사용하고, 단순한 경우에만 ASCII `text` 블록을 쓴다.
- **`## 핵심 코드 조각`** — 2~8줄 snippet + "이 조각으로 알 수 있는 것" bullet 2~3개.
- **`## 실제로 해보는 예시`** — 입력/호출/클릭과 관찰 결과를 bullet로 분리.
- **`## 데모/시연 스크립트`** — 고객에게 보여줄 화면/호출 순서와 말할 포인트를 분리한다.
- **`## 현장 체크리스트`** — 체크박스 bullet.
- **`## 근거 지도`** — 코드 경로와 line reference를 모으는 기본 위치. `먼저 볼 소스 지도`가 필요하면 이 섹션을 표로 확장한다.

언제 쓰나:

- 처음 보는 사람이 "이 단어가 뭔지"에서 멈출 가능성이 높다.
- 코드 근거가 3개 이상 파일을 건넌다.
- UI 비노출, route 접근, API 권한처럼 서로 다른 표면을 구분해야 한다.
- 고객사 현장에서 같은 판단을 반복할 가능성이 있다.

주의:

- snippet은 근거의 전부가 아니다. 정확한 원본 위치는 `근거 지도`에 둔다.
- 실제 예시는 production 동작을 과장하지 않는다. "열릴 수 있음", "별도 확인 필요", "막혀야 함"처럼 조건을 bullet로 분리한다.
- 초심자용 설명을 넣더라도 일반 튜토리얼로 만들지 않는다. 핵심은 GenOS에서 다시 써먹을 패턴·함정·판단이다.
- 장문 설명이 필요하면 먼저 고객/현장 관점의 결론을 쓰고, 그 다음에 bullet/diagram으로 보충한다.
- "줄이기"가 목표가 아니다. 길어져도 스캔 가능하면 성공이고, 짧아져도 판단 근거가 사라지면 실패다.

파일명: 순수 제목 `.md`. GenOS 플랫폼 노트는 vault 관행대로 `GenOS ` 접두어 포함(예: `GenOS admin-api 외부 시스템 로그 연동 패턴.md`). 고객사 노트는 고객사명으로 시작(예: `삼성카드 PIMS 행위로그 연동.md`).

원자성: **한 노트 = 한 아이디어.** 쪼갤 수 있으면 쪼갠다. 대화의 번호 항목을 옮길 땐 일부를 압축·생략하지 말고 동일 깊이로.

### 5. MOC 갱신 (생략 금지)
`<vault>/Projects/GenonAI/GenOS/GenOS 지식 MOC.md`를 열어 적절한 섹션(코드 온보딩 / 운영·장애 / 패턴·함정 / 고객사별 / 학습)에 `- [[새 노트]] — 한 줄 요지` 추가. 새 노트의 frontmatter `related_notes`에도 `[[GenOS 지식 MOC]]`를 넣어 양방향으로 엮는다.

### 6. 결과 보고
작성/갱신한 노트 목록과 MOC 변경을 요약한다.

## Gotchas — 여기서 실제로 틀린다

이 스킬을 실행할 때 빠지기 쉬운 함정들. 규칙이 아니라 *왜 틀리는지*로 기억하라.

- **MOC 갱신을 빼먹는다 (1번 실패 모드).** 노트만 쓰고 만족하기 쉽다. 하지만 MOC에 링크를 안 걸면 새 세션은 그 노트의 존재 자체를 모른다 — AI는 그래프뷰를 못 보고 MOC만 진입점으로 읽기 때문이다. *작성한 노트는 반드시 MOC에 한 줄 추가하고 끝낸다.*
- **중복 노트를 새로 만든다.** 비슷한 주제가 이미 있는데 검색을 건너뛰고 새 파일을 만들면, 지식이 두 곳에 갈라져 둘 다 신뢰를 잃는다. 작성 전 `grep -rl`로 먼저 찾고, 있으면 갱신한다.
- **세션 상태를 지식으로 착각한다.** "오늘 chat_service 3줄 고쳤다"는 패턴이 아니라 상태다. "6개월 뒤 다른 고객사에서 시간을 아껴줄까?"를 통과 못 하면 캡처하지 말고 HANDOFF로 보낸다.
- **존재하지 않는 노트를 `[[링크]]`한다.** related_notes의 모든 wikilink는 vault에 실존하는 노트 + 이번에 함께 만드는 노트여야 한다. 미존재 링크는 깨진 그래프를 만든다.
- **고객사 노트 위치가 혼재한다.** 현재 삼성카드 노트는 `GenonAI/` loose에 있다. 한 고객사 노트가 3장 이상 쌓이면 `GenonAI/<고객사>/` 폴더 분리를 *제안*하되, 기존 파일 이동은 사용자 확인 후에만 — 임의로 옮기면 다른 노트의 `[[링크]]`가 깨질 수 있다.
- **회사 자산을 건드린다.** 캡처 자동화가 아쉬워도 회사 레포 `.claude/skills/genos-*`는 절대 수정하지 않는다. 모든 자동화는 이 개인 스킬(`~/.claude/`) 안에서 끝낸다.
- **근거 경로가 본문을 잡아먹는다.** GenOS 노트는 코드 근거가 많아서 설명문에 `file:line`이 섞이면 읽는 사람이 핵심을 놓친다. 고객 설명과 판단은 본문에, 경로는 `근거 지도`로 분리한다.
- **story-first를 장문 에세이로 착각한다.** story-first는 고객이 이해할 흐름을 먼저 잡는다는 뜻이지, bullet/table/diagram을 버린다는 뜻이 아니다. 기존 장문의 내용은 보존하고, 읽는 순서만 현장 설명에 맞게 재배치한다.
- **Mermaid를 장식으로 넣는다.** diagram은 하나의 질문에 답해야 한다. 책임 경계, 요청 순서, 상태 변화, 데이터 관계를 줄여주지 못하면 텍스트나 표가 낫다.

## 예시

**예시 1 — 플랫폼 패턴 캡처**
세션에서 "외부 시스템 전송은 BackgroundTasks + dry-run 플래그로 한다"는 패턴을 적용함.
→ `GenOS/GenOS admin-api 외부 시스템 로그 연동 패턴.md` (tags: `type/pattern`, `backend/integration`), MOC "패턴·함정"에 링크.

**예시 2 — 고객사 결정 캡처**
"삼성카드 고객관리번호는 CST_MNGT_NO로 넘어오고, 컬럼은 삼성카드 DB에만 추가"가 확정됨.
→ `삼성카드 PIMS 행위로그 연동.md` 기존 노트 **갱신**(중복 생성 X), tags `customer/samsung-card` `feature/pims`.

**예시 3 — 캡처 거부**
"오늘 chat_service_service.py 3줄 고치고 커밋함" → 세션 상태. **캡처하지 않고** HANDOFF로 안내.
