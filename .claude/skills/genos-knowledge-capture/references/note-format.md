# GenOS 지식 노트 작성 형식

이 문서는 `genos-knowledge-capture`가 노트를 **실제로 쓸 때**의 형식 상세다. SKILL.md 워크플로우 4단계(노트 작성)에서 이 파일을 읽고 따른다. SKILL.md에는 워크플로우와 판단 기준만 두고, 형식 스펙은 여기로 분리했다 — skill 트리거 시 매번 컨텍스트에 올리지 않기 위해서다.

**1차 독자는 "이 기술을 처음 보는 사람"이다.** 고객 미팅용 대본이 아니라, 입사 직후의 동료나 다음 세션의 AI가 *이 문서 한 장만으로* 개념을 잡고 "그래서 GenOS에서 이게 어디에 있는지"까지 이해할 수 있게 쓴다. 자기완결(self-contained)이 목표다.

## Frontmatter

```yaml
---
source:
  - "admin-api/src/main.py"               # repo-relative evidence identifier
  - "origin/project/samsung-card/feat/96:admin-api/src/util/pims_utils.py"
  - "SAMSUNGCARD-96"
related_notes:
  - "[[실존하는_노트만]]"                  # 반드시 vault에 존재하는 노트 + 이번에 함께 만드는 노트
tags:
  - work/genos
  - type/pattern        # 또는 type/moc, type/onboarding
  - customer/samsung-card  # 고객사 노트일 때
  - feature/<기능>         # 횡단 검색용
created: YYYY-MM-DD        # 오늘 날짜
---
```

`source`는 Obsidian 클릭 링크가 아니라 **근거 식별자**로 쓴다. URI-style prefix나 개인 PC 절대경로를 붙이지 않는다.
- GenOS 레포 파일: `admin-front/src/layouts/UserLayout.js`처럼 repo-relative path.
- 특정 브랜치 파일: `origin/project/samsung-card/feat/96:admin-api/src/util/pims_utils.py`처럼 `git show <branch>:<path>`에 바로 넣는 형태.
- 다른 repo가 근거면: `genos-chart/helmfile.yaml`처럼 repo 이름을 첫 segment로.
- 이슈: `SAMSUNGCARD-96`처럼 사람이 바로 읽는 식별자.
- 본문 근거는 필요하면 `admin-front/src/layouts/UserLayout.js:95`처럼 repo-relative `file:line`으로 인용.

## 본문 spine — 담백한 기능형 8섹션

기본 골격이다. 제목은 글쓴이의 상황(고객 미팅 등)이 아니라 **독자가 찾는 정보**를 가리키는 담백한 기능명으로 쓴다.

1. `## 한 줄 결론` — 이 노트에서 가장 먼저 기억할 판단 1문장.
2. `## 먼저 알아야 할 개념` — 자기완결의 핵심 섹션 (아래 별도 상세).
3. `## 무엇이고 왜 있나` — 이 기능/구조가 무엇이고, 없으면 현장에서 무엇이 막히는지(운영·보안·분석 문제).
4. `## 큰 그림` — 첫 Mermaid 또는 ASCII diagram. 책임 경계와 주요 경로를 한눈에. 바로 아래 "이 그림에서 봐야 할 점" bullet 2~4개.
5. `## 어떻게 동작하나` — 런타임 흐름, API 흐름, 배치 흐름을 필요한 만큼 diagram + bullet로.
6. `## 헷갈리는 지점` — 비슷한 용어 구분, 흔한 오해 교정, "목록 개수 ≠ 실제 리소스 개수" 류. 사용자가 재진술한 문장의 맞는 부분/틀린 부분을 분리해 고친다.
7. `## 코드·근거` — 코드 레벨 구조(enum·schema·DB model·helper pattern), snippet 2~8줄 + "이 조각으로 아는 것" bullet, 그리고 근거 경로(`file:line`)를 모은다.
8. `## 다음에 확인할 것` — 구체적 후속 질문. 키워드 나열 금지.

**모든 노트가 8개를 강제로 갖지 않는다.** 짧은 고객사 결정 노트나 단일 함정 노트는 핵심만 남긴다. 단 입문자 자기완결의 두 기둥인 `## 먼저 알아야 할 개념`과 `## 코드·근거`는 웬만하면 유지한다.

## `## 먼저 알아야 할 개념` — 자세히

이 노트를 이해하는 데 필요한 선행지식을, **처음 보는 사람 기준**으로 깐다. 자기완결의 핵심이다.

- **길어도 좋다.** 분량 제한은 없다. 짧게 압축하기보다, 개념이 왜 그렇게 생겼고 GenOS에서 어떻게 쓰이는지 *서사로* 풀어 읽는 재미를 만든다. 입문자는 건조한 정의 나열보다 "왜 이렇게 됐나" 이야기에서 더 잘 따라온다.
- **단, 일반 튜토리얼로 새지 않는다.** 판정 기준은 분량이 아니라 **착지**다:
  > *"이 문단이 GenOS와 무관하게도 그대로 성립하면 위키 복붙이다 — 잘라내거나 GenOS 맥락으로 당겨온다."*

  모든 개념은 결국 "그래서 GenOS/이 노트에선 ○○" 으로 착지해야 한다.
- **고르는 기준**: 모르면 이 노트가 막히는 개념만 넣는다. 알아도 그만인 건 뺀다.
- **순서**: 일반 개념 → GenOS 용어 매핑 → 실제 개수/흐름 검증 → 비슷한 용어 분리 → (필요시) 사용자 표현을 더 정확한 문장으로 교정.

예시 (길게, 서사형 — 이 정도 길이를 권장):

```
Kubernetes를 처음 본다면 Pod와 PVC부터 잡자. Pod는 컨테이너 하나(혹은 몇 개)를
묶어 실행하는 최소 단위다 — "프로세스 한 묶음을 띄운 것"으로 보면 된다. PVC
(PersistentVolumeClaim)는 그 Pod에 붙이는 디스크 청구서다. Pod는 죽으면 사라지지만
PVC에 쓴 데이터는 남는다. 그래서 쿠버네티스는 "실행(Pod)"과 "저장(PVC)"을 일부러
분리한다 — 실행 단위는 언제든 갈아끼우고, 데이터는 그대로 두기 위해서다.

GenOS는 이 둘을 자기 용어로 감싼다. Codespace는 시작하면 뜨는 개발용 Pod이고,
CodespaceVolume이 거기 붙는 PVC다. 이 매핑을 알면 이 노트의 핵심 — "코드스페이스
100개가 등록돼 있어도 실행 중이 0개면 Pod는 0개" — 가 자연스럽게 읽힌다. 등록은
DB row이고 실행이 Pod이기 때문이다. 둘을 같은 것으로 보면 리소스 추정이 어긋난다.
```

이 예시는 길지만 한 문단도 "일반 K8s 설명"으로 끝나지 않고 매번 GenOS(Codespace/CodespaceVolume)로 착지한다 — 그래서 튜토리얼이 아니다.

이 패턴이 특히 필요한 주제:
- Kubernetes 리소스가 얽힌 개념: Pod, Deployment, Service, PVC, VirtualService, AuthPolicy.
- GenOS 안에서 이름이 비슷한 기능: 코드스페이스/코드서빙, 워크플로우/서비스 라우터, 모델서빙/코드서빙.
- "목록 개수 = 실제 리소스 개수?"처럼 DB row·설정·런타임 리소스를 구분해야 하는 질문.

## Mermaid 원칙

- Mermaid는 **1개로 제한하지 않는다.** 각 diagram은 하나의 질문에 답해야 한다.
- 텍스트 100마디보다 도표 하나가 빠르면 적극 사용한다. 장식용 diagram은 넣지 않는다.
- 첫 diagram(`## 큰 그림`)은 전체 맥락을 잡는 용도. 이후 diagram은 runtime sequence, branching, lifecycle, data relationship처럼 섹션별 이해 병목을 풀 때 추가한다.
- `flowchart`는 책임 경계/분기, `sequenceDiagram`은 요청 순서, `stateDiagram`은 상태 변화, `erDiagram`은 데이터 관계, `graph`는 시스템 구성요소 관계.
- Mermaid 아래엔 "이 그림에서 봐야 할 점" bullet 2~4개를 붙인다.
- 밝은 배경 노드는 반드시 `color:#000` 지정(미지정 시 흰 글씨로 안 보임). 줄바꿈은 `\n`이 아니라 `<br>`. 점선 라벨에 `.`(예: `7.24`)을 넣으면 화살표 종료로 오인되니 `7/24`로 쓴다.

## 문체 규칙

- 첫머리는 사람이 읽는 문장으로 열고, 판단·분기·함정은 bullet·표·diagram으로 스캔 가능하게 만든다.
- **정보량은 보존한다.** 줄여야 할 땐 세션 상태/중복/일회성 정보만 제거하고, 판단 근거·예외·trade-off는 남긴다. "줄이기"가 목표가 아니다 — 길어져도 스캔 가능하면 성공, 짧아져도 판단 근거가 사라지면 실패.
- 일반 문단은 4~6문장 권장. `## 먼저 알아야 할 개념`처럼 설명형 섹션은 더 길어도 좋다. 그 외 섹션이 길어지면 bullet/diagram으로 쪼갠다.
- bullet은 "소제목 → 설명 → 근거 → 주의"로 깊어져도 된다. 단 각 줄은 한 판단만 담는다.
- 소스 경로가 문장의 주어가 되면 가독성이 떨어진다. 판단을 먼저 쓰고 경로는 뒤에 붙인다. `file:line` 근거가 3개 이상이면 본문에 흩뿌리지 말고 `## 코드·근거`로 모은다.
- 표는 "분류/판단"에만 쓴다. 긴 설명을 표 안에 우겨 넣지 않는다.
- 실제 동작 예시는 production을 과장하지 않는다. "열릴 수 있음", "별도 확인 필요", "막혀야 함"처럼 조건을 분리한다.

## 섹션을 더 추가할 때 (조건)

기본 8섹션으로 부족할 때만 추가한다. 아래는 추가 후보다.

- `## 읽는 모드` — 입문자 10분 / 숙련자 2분 참조처럼 독자의 읽는 깊이를 안내해야 할 때.
- `## 이 노트에서 쓰는 말` — 처음 보는 단어가 많을 때, 한 줄 정의 표.
- `## 실제로 해보는 예시` — 입력/호출/클릭과 관찰 결과를 bullet로 분리.
- `## 데모/시연 스크립트` — 고객·동료에게 보여줄 화면/호출 순서와 말할 포인트.
- `## 현장 체크리스트` — 작업 전 확인할 항목을 체크박스로.

언제 이 추가가 필요한가:
- 처음 보는 사람이 용어에서 멈출 가능성이 높을 때.
- 코드 근거가 3개 이상 파일을 건널 때.
- UI 비노출·route 접근·API 권한처럼 서로 다른 표면을 구분해야 할 때.
- 고객사 현장에서 같은 판단을 반복할 가능성이 있을 때.

## 파일명 · 원자성

- 파일명: 순수 제목 `.md`. GenOS 플랫폼 노트는 vault 관행대로 `GenOS ` 접두어 포함(예: `GenOS admin-api 외부 시스템 로그 연동 패턴.md`). 고객사 노트는 고객사명으로 시작(예: `삼성카드 PIMS 행위로그 연동.md`).
- 원자성: **한 노트 = 한 아이디어.** 쪼갤 수 있으면 쪼갠다. 대화의 번호 항목을 옮길 땐 일부를 압축·생략하지 말고 동일 깊이로 옮긴다.
