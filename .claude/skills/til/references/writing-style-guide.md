# 작성 스타일 가이드

이 가이드는 TIL 문서를 "지식 전달"의 관점에서 작성하는 방법을 다룬다. 규칙의 나열이 아니라, 좋은 TIL 문서가 만들어지는 흐름을 처음부터 끝까지 따라간다.

독자는 제목을 보고 클릭하고, 한 줄 설명으로 기대감을 갖고, 결론에서 전체 그림을 파악한 뒤, 본문의 이야기를 따라가며 이해를 쌓고, 마지막 정리에서 핵심을 기억에 새긴다. 이 가이드는 그 여정의 각 단계를 다룬다.

---

## 1. 제목: 첫 인상이 전부다

독자가 TIL 목록에서 제목을 보고 클릭할지 말지를 결정한다. 제목은 단순한 라벨이 아니라 "이 글을 왜 읽어야 하는가"에 대한 첫 번째 답이다.

### 좋은 제목의 패턴

| 패턴 | 예시 | 왜 좋은가 |
|------|------|----------|
| **통념을 뒤집는** | "Node.js가 싱글스레드라는 미신" | 독자가 알고 있던 것이 틀렸다는 충격 |
| **혼동되는 개념 비교** | "Concurrency와 Parallelism - 동시성과 병렬성" | 두 개념의 차이를 알고 싶게 만듦 |
| **질문 형태** | "왜 Spring은 CGLIB 프록시를 기본으로 선택했을까?" | 직접적인 호기심 자극 |
| **구체적 발견** | "Python의 f-string이 format()보다 빠른 진짜 이유" | "진짜 이유"가 궁금하게 만듦 |

```markdown
Bad: 평범한 라벨
"Node.js 이벤트 루프"
"동시성 개념 정리"
"DI 패턴"

Good: 호기심을 유발하는 제목
"Node.js가 싱글스레드라는 미신"
"왜 Spring은 모든 Bean을 싱글톤으로 만들까?"
"Python의 GIL - 왜 멀티스레딩이 느릴까?"
```

**핵심 원칙:** 제목을 읽은 독자가 "어? 정말?" 또는 "왜지?"라고 반응해야 한다. 그 반응이 없으면 제목을 다시 쓰라.

---

## 2. 한 줄 설명: 호기심의 문을 열어라

제목이 시선을 끌었다면, 바로 아래의 한 줄 설명이 독자의 호기심을 구체적으로 만든다. 가장 효과적인 방법은 **질문**이다. 질문은 독자의 뇌에 "미완성의 빈칸"을 만들고, 사람은 본능적으로 그 빈칸을 채우고 싶어한다.

```markdown
Bad: 수동적 설명
"Concurrency와 Parallelism에 대해 정리한다."
"AI Agent의 개념을 알아본다."

Good: 질문으로 호기심 유발
"'동시성'과 '병렬성'은 같은 말일까? 왜 이 두 개념을 구분해야 할까?"
"AI Agent의 개념, 역사, 그리고 왜 지금 '에이전트 시대'라고 불리는지"
```

질문형이 어렵다면, **범위 + 핵심 의문**을 조합하라. "X의 개념, Y, 그리고 왜 Z인지"라는 패턴이 효과적이다. 독자는 이 한 줄을 읽고 "이 질문의 답이 궁금하다"고 느껴야 한다. 그래야 스크롤을 내린다.

---

## 3. "결론부터 말하면": 1분 안에 핵심을 전달하라

이 섹션이 문서 전체에서 가장 중요하다. 독자가 시간이 없어서 이 섹션만 읽고 나간다고 가정하라. 그래도 핵심을 전달할 수 있어야 한다.

**왜 결론이 먼저인가?** 인간의 뇌는 전체 그림을 먼저 파악해야 세부사항을 효과적으로 흡수한다. 결론을 미리 제시하면 독자는 이후의 설명을 "확인"하는 자세로 읽게 되어, 이해도와 기억 유지율이 높아진다. 지도를 먼저 보고 여행을 시작하는 것과 같다.

### 구성: 핵심 요약 + 시각적 보조

핵심 요약 2-3문장으로 결론을 제시하고, 바로 아래에 다이어그램이나 비교 테이블을 배치하라. 글만으로 전달하기 어려운 구조나 관계를 시각적으로 보여주면, 독자가 결론을 직관적으로 이해하게 된다.

```markdown
## 결론부터 말하면

**Node.js는 싱글스레드가 아니다.** "JavaScript 실행"만 단일 스레드일 뿐,
내부적으로는 멀티스레드를 적극 활용한다.

[mermaid 다이어그램: JavaScript 실행 vs 내부 스레드 구조]

| 구분 | 스레드 | 처리 방식 |
|------|--------|----------|
| JavaScript 실행 | **단일** | V8 엔진 |
| 파일 I/O | **멀티** | libuv Thread Pool |
| DNS | **멀티** | libuv Thread Pool |
| 네트워크 I/O | **단일** | OS 비동기 API (epoll/kqueue) |
```

여기서 핵심 요약의 첫 문장은 **이 문서의 제목에 대한 직접적인 답**이어야 한다. 제목이 "Node.js가 싱글스레드라는 미신"이라면, 결론의 첫 문장은 "Node.js는 싱글스레드가 아니다"가 되어야 한다. 제목의 약속을 결론에서 즉시 이행하라.

## 4. "왜(Why)"를 반드시 설명하라

결론을 제시했으면, 이제 "왜 그런지"를 설명할 차례다. 이것이 TIL 문서의 핵심이다. "무엇(What)"과 "어떻게(How)"만 설명하면 금방 잊혀진다. "왜(Why)"를 이해하면 맥락이 생기고, 맥락이 있는 지식은 기억에 오래 남는다.

| 질문 | 설명 |
|------|------|
| **왜 이 기술이 필요한가?** | 이 기술이 없으면 어떤 문제가 생기는지 |
| **왜 이렇게 설계되었는가?** | 다른 방법 대신 이 방법을 선택한 이유 |
| **왜 이런 결과가 나오는가?** | 내부 동작 원리 |

```markdown
Bad: 정의만 나열
"DI는 의존성을 외부에서 주입하는 패턴입니다."
"Node.js는 싱글스레드 기반 이벤트 루프입니다."

Good: "왜?"로 시작
"만약 프레임워크 없이 웹 서버를 만든다면? 우리가 직접 해야 할 일들이 산더미처럼 쌓인다..."
"만약 정말 싱글스레드라면, 파일 4개를 동시에 읽는 게 어떻게 가능할까?"
```

"왜"를 설명하는 가장 자연스러운 방법은 **문제 상황에서 출발하는 것**이다. "이 기술이 없던 시절에는 어떤 고통이 있었는가?"를 보여주면, 독자는 자연스럽게 "그래서 어떻게 해결했는데?"라고 궁금해한다. 그 순간이 바로 본문으로 넘어가는 다리가 된다.

## 5. 스토리텔링으로 풀어가라

"왜"라는 질문을 던졌으면, 이제 그 질문에 대한 답을 이야기로 풀어가야 한다. 이것은 단순한 문체 선호가 아니다. 인간의 뇌는 이야기 구조에서 정보를 가장 잘 처리한다. "사실의 나열"보다 "문제 -> 긴장 -> 해결"의 서사가 훨씬 강력하게 기억에 남는다.

### 스토리텔링 패턴

| 패턴 | 설명 | 예시 |
|------|------|------|
| **문제 -> 의문 -> 해답** | 문제 상황을 먼저 보여주고, 의문을 제기한 후, 해답 제시 | "싱글스레드인데 어떻게 동시에? -> libuv가 있기 때문" |
| **만약 ~라면?** | 가정을 통해 문제를 체감하게 함 | "만약 Spring 없이 웹 서버를 만든다면?" |
| **이상한 점 발견** | 당연하게 알던 것에 의문 제기 | "근데 이상하다. 싱글스레드라면서 왜 4개가 동시에?" |
| **역사적 맥락** | 기술이 왜 등장했는지 배경 설명 | "2005년까지 개발자들은 공짜 점심을 즐겼다..." |

이 네 패턴은 모두 같은 원리를 공유한다. **독자의 머릿속에 "빈칸"을 만들고, 그 빈칸을 채워가는 과정이 본문이 된다.** "만약 ~라면?"은 가정으로 빈칸을 만들고, "이상한 점 발견"은 모순으로 빈칸을 만들고, "역사적 맥락"은 과거와 현재의 간극으로 빈칸을 만든다.

### 연결어로 흐름을 만들어라

이야기에는 흐름이 있어야 한다. 문장과 문장 사이, 단락과 단락 사이에 연결어를 적극 사용하라. 연결어는 독자에게 "지금 이야기가 어디로 가고 있는지"를 알려주는 이정표다.

```markdown
Good: 자연스러운 흐름
"하지만 여기서 문제가 생겼다."
"왜일까?"
"그렇다면 이건 어떻게 설명할 수 있을까?"
"이제 답이 보인다."
"그래서 libuv가 등장한 것이다."

Bad: 단편적 나열
"libuv는 Thread Pool을 가진다."
"Thread Pool은 기본 4개이다."
"파일 I/O는 Thread Pool을 사용한다."
```

Good 예시는 하나의 이야기처럼 읽힌다. "문제 -> 의문 -> 힌트 -> 해결"의 호흡이 느껴진다. Bad 예시는 사전의 항목처럼 읽힌다. 같은 정보를 담고 있지만, 독자의 뇌에 미치는 영향은 완전히 다르다.

### 문단 단위로 설명하라

한 줄씩 끊어서 쓰면 기술 사양서가 된다. 문단으로 풀어서 쓰면 이야기가 된다. TIL은 이야기여야 한다.

```markdown
Bad: 한 줄씩 끊김
- libuv는 비동기 I/O 라이브러리다.
- Thread Pool을 가진다.
- 기본 4개다.

Good: 문단으로 풀어서
libuv는 Node.js의 비동기 I/O를 담당하는 C 라이브러리다. Ryan Dahl이 Node.js를 만들 때
직접 개발했다. 왜 필요했을까? JavaScript 자체에는 파일을 읽거나 네트워크 통신을 하는
기능이 없다. 브라우저에서는 브라우저가 이런 기능을 제공하지만, 서버에서는 Node.js가
직접 제공해야 한다. 그 역할을 libuv가 한다.
```

Good 예시를 보면, 사실 -> 배경 -> 질문 -> 답변이 하나의 문단 안에서 자연스럽게 이어진다. 문단 하나를 읽었을 뿐인데, 독자는 "libuv가 뭔지", "왜 필요한지", "어떤 역할인지"를 모두 이해하게 된다.

### 스토리텔링 전체 흐름 예시

위의 원칙들이 실제로 어떻게 결합되는지 보여주는 전체 흐름이다:

```markdown
## 1. 왜 "싱글스레드"라는 오해가 생겼을까?

### 1.1 개발자가 보는 세계

Node.js로 코드를 작성할 때, 우리는 멀티스레드를 전혀 의식하지 않는다.
`new Thread()`도 없고, `synchronized`도 없다. 그냥 JavaScript를 쓸 뿐이다.

이렇게 개발자 입장에서는 모든 것이 단일 스레드로 보인다.
그래서 "Node.js는 싱글스레드"라고 생각하기 쉽다.

### 1.2 그런데 이상한 점이 있다

만약 정말 싱글스레드라면, 이 코드는 어떻게 동작해야 할까?

[코드 예시]

싱글스레드라면 순차적으로 읽어야 한다. 각 파일이 10ms 걸린다면 총 40ms가 걸려야 정상이다.

**하지만 실제 출력은?**

[출력 결과]

거의 동시에 완료된다! 어떻게 싱글스레드에서 이게 가능할까?

### 1.3 비밀은 libuv에 있다

[해답 설명]
```

이 흐름을 보면: "개발자가 보는 세계"(공감) -> "이상한 점"(의문) -> "해답"(해결)으로 이어진다. 독자는 자연스럽게 "나도 그렇게 생각했는데?" -> "어, 정말 이상하네?" -> "아, 그래서 그랬구나!"라는 감정의 흐름을 경험한다. 이것이 스토리텔링의 힘이다.

## 6. 코드 예시: 최소한으로, 비교로, 주석으로

코드는 TIL의 핵심 증거물이다. 글로 설명한 것을 코드가 증명한다. 하지만 장황한 코드는 오히려 이해를 방해한다. 독자는 코드를 "읽는" 것이 아니라 "스캔"한다. 핵심이 한눈에 들어와야 한다.

### 세 가지 원칙

**첫째, 최소한의 코드.** 핵심을 보여주는 데 필요한 코드만 쓰라. import, try-catch, 보일러플레이트는 과감하게 생략하라. 독자가 보고 싶은 것은 "이 기술이 어떻게 동작하는가"이지, "이 코드를 실행하려면 뭐가 필요한가"가 아니다.

```markdown
Bad: 장황한 보일러플레이트
import java.util.concurrent.CompletableFuture;
import java.util.concurrent.ExecutionException;

public class AsyncExample {
    public static void main(String[] args) throws Exception {
        CompletableFuture<String> f1 = CompletableFuture.supplyAsync(() -> {
            try {
                return fetchFromAPI("service1");
            } catch (Exception e) {
                throw new RuntimeException(e);
            }
        });
        // ... 50줄 더
    }
}

Good: 핵심만
CompletableFuture<String> f1 = CompletableFuture.supplyAsync(() -> fetchFromAPI("service1"));
CompletableFuture<String> f2 = CompletableFuture.supplyAsync(() -> fetchFromAPI("service2"));
CompletableFuture.allOf(f1, f2).join();  // 동시에 대기
```

**둘째, Before/After 비교.** 코드를 하나만 보여주면 "그래서 뭐가 좋은데?"라는 질문에 답할 수 없다. 나란히 비교해야 "왜 이렇게 하는지"가 체감된다.

```java
// Bad: 순차 처리 - 총 3초 (CPU는 대부분 놀고 있음)
String data1 = fetchFromAPI("service1");  // 1초 대기
String data2 = fetchFromAPI("service2");  // 1초 대기
String data3 = fetchFromAPI("service3");  // 1초 대기

// Good: Concurrent 처리 - 총 ~1초 (대기 시간 중첩)
CompletableFuture<String> f1 = CompletableFuture.supplyAsync(() -> fetchFromAPI("service1"));
CompletableFuture<String> f2 = CompletableFuture.supplyAsync(() -> fetchFromAPI("service2"));
CompletableFuture<String> f3 = CompletableFuture.supplyAsync(() -> fetchFromAPI("service3"));
CompletableFuture.allOf(f1, f2, f3).join();
```

주목할 점: 첫 줄의 주석에 **"총 3초"와 "총 ~1초"**라는 결과를 미리 써넣었다. 독자는 코드를 분석하기 전에 이미 "3배 빨라진다"는 것을 알게 된다. 이것도 "결론부터 말하면"의 원칙이다.

**셋째, 인라인 주석으로 핵심을 설명하라.** 코드 옆에 "왜"를 적어라. 독자가 코드와 본문 설명을 왔다갔다 하지 않아도 되게 하라.

```java
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 10_000; i++) {
        executor.submit(() -> {
            String result = fetchFromAPI();  // 블로킹처럼 보이지만
            process(result);                  // 실제론 논블로킹으로 동작
        });
    }
}   // 10,000개의 가상 스레드 생성 - 메모리 단 수십 MB
```

이 세 가지 원칙을 요약하면: **짧게, 비교하고, 설명하라.**

## 7. 스타일 규칙

지금까지가 "무엇을 쓸 것인가"의 문제였다면, 여기서부터는 "어떻게 쓸 것인가"의 문제다. 스타일 규칙은 작아 보이지만, 일관성 있게 지키면 문서의 전문성과 가독성이 크게 높아진다.

### 언어

- **설명**: 한국어
- **코드/기술 용어**: 영어 (DI, IoC, Bean, Scope 등)

### Bold 처리 규칙 (IMPORTANT)

Bold는 핵심 강조에 사용한다. 닫는 `**` 다음에 반드시 띄어쓰기가 있어야 한다.

일부 마크다운 파서에서 닫는 `**` 바로 다음에 한글/영문이 붙으면 bold가 적용되지 않는다.

```markdown
Bad: 닫는 ** 다음에 띄어쓰기 없음
**Kubernetes 1.29부터 Native Sidecar가 정식 지원(Stable)**된다.

Good: 닫는 ** 다음에 띄어쓰기
**Kubernetes 1.29부터 Native Sidecar가 정식 지원(Stable)** 된다.
```

```markdown
Bad: 닫는 ** 다음에 바로 조사가 붙음
**Concurrency(동시성)**와 **Parallelism(병렬성)**은

Good: 닫는 ** 다음에 띄어쓰기 (또는 문장 구조 변경)
**Concurrency(동시성)** 와 **Parallelism(병렬성)** 은
또는: Concurrency(**동시성**)와 Parallelism(**병렬성**)은
```

> **참고:** 띄어쓰기가 어색할 경우, bold 범위를 조정하거나 문장 구조를 바꾸는 것도 방법이다.

```markdown
Bad: 콜론 뒤 내용까지 bold
**Step 1: 설명 (보충)**

Good: 핵심만 bold
**Step 1:** 설명 (보충)
```

### 비교 표현

비교는 이해를 돕는 가장 강력한 도구다. 상황에 따라 적절한 형식을 선택하라:

**테이블:** 여러 항목을 체계적으로 비교할 때
```markdown
| 특징 | Python | Java |
|------|--------|------|
| 타입 | 동적 | 정적 |
```

**코드 비교:** 같은 로직의 구현 차이를 보여줄 때
```markdown
# Python
def example():
    pass

# Java
public void example() { }
```

**Do's and Don'ts:** 올바른 방법과 잘못된 방법을 대비할 때
```markdown
Good: 이렇게 하라
Bad: 이렇게 하지 마라
```

### 수식 표현 (LaTeX)

수학 수식은 LaTeX 문법으로 작성한다. 일반 텍스트로 수식을 쓰면 가독성이 떨어지고 정확성도 떨어진다.

```markdown
Bad: 일반 텍스트로 수식 작성
S = 1 / ((1 - P) + P/N)

Good: LaTeX 인라인 수식
$S = \frac{1}{(1-P) + \frac{P}{N}}$

Good: LaTeX 블록 수식
$$
S = \frac{1}{(1-P) + \frac{P}{N}}
$$
```

**인라인 vs 블록 구분:**

| 상황 | 형식 | 예시 |
|------|------|------|
| 문장 중간에 자연스럽게 | **인라인** `$...$` | B-Tree는 **항상** $O(\log n)$을 보장한다. |
| 독립적인 공식, 강조 필요 | **블록** `$$...$$` | 아래 공식을 보면... |
| 여러 줄의 복잡한 수식 | **블록** `$$...$$` | 증명이나 유도 과정 |

**LaTeX 사용 시점:**
- 분수, 제곱, 루트 등 복잡한 수식
- 그리스 문자 ($\alpha$, $\beta$, $\infty$ 등)
- 수학적 정리나 공식
- Big-O 표기법 ($O(n)$, $O(\log n)$ 등)

### 출처 표기

문서 맨 마지막에 `## 출처` 섹션 추가. 공식 문서를 가장 위에 배치한다.

```markdown
## 출처

- [Spring Framework Documentation](https://docs.spring.io/...) - 공식 문서
- [블로그 제목](https://blog.example.com/article)
```

출처는 단순한 형식 요구가 아니다. 독자가 더 깊이 파고들고 싶을 때 안내하는 역할을 한다. 공식 문서를 최상단에 두는 이유는, 가장 정확한 정보원으로 안내하기 위해서다.

## 8. "정리" 섹션: 기억에 남기는 마무리

문서의 마지막 "정리" 섹션은 독자가 이 문서에서 가져갈 핵심을 명확하게 새기는 곳이다. 독자가 1주일 후에 이 문서를 다시 펼치면, "결론부터 말하면"과 "정리" 두 섹션만 다시 읽을 것이다. 그만큼 이 섹션의 밀도가 중요하다.

### 구조: 번호 목록 + 핵심 키워드 Bold + 1-2줄 설명

```markdown
## 정리

### 핵심 포인트

1. **Concurrency =/= Parallelism**
   - Concurrency: 여러 일을 **다루는** 것 (구조)
   - Parallelism: 여러 일을 **동시에 하는** 것 (실행)

2. **왜 구분이 중요한가**
   - Concurrent하게 설계해야 Parallel 실행의 혜택을 받을 수 있다
   - I/O bound -> Concurrency, CPU bound -> Parallelism

3. **현대 시스템의 필수 요소**
   - CPU 클럭 한계 -> 멀티코어 시대
   - Concurrency 없이는 하드웨어 잠재력을 활용할 수 없다
```

### 세 가지 원칙

**3-5개로 제한하라.** 핵심 포인트가 7개, 10개가 되면 아무것도 기억에 남지 않는다. "이 문서에서 딱 3가지만 기억한다면?"이라는 질문에 답하는 셈으로 쓰라.

**각 포인트는 독립적으로 읽혀야 한다.** 정리 섹션만 읽는 독자도 각 포인트를 이해할 수 있어야 한다. 본문을 읽지 않은 사람도 "아, 이런 이야기구나"를 감잡을 수 있는 수준이면 된다.

**첫 번째 포인트가 가장 중요하다.** "이 문서에서 딱 하나만 기억한다면?"이라는 질문의 답이 첫 번째 포인트여야 한다. 위 예시에서는 "Concurrency와 Parallelism은 다른 개념이다"가 그것이다.

## 전체 흐름 요약

좋은 TIL 문서는 독자를 하나의 여정으로 이끈다:

```
제목 (호기심)
  -> 한 줄 설명 (궁금증 구체화)
    -> 결론부터 말하면 (전체 지도)
      -> 왜? (문제 상황)
        -> 스토리텔링 (이야기로 풀기)
          -> 코드 예시 (증거)
            -> 정리 (기억에 새기기)
              -> 출처 (더 깊이)
```

각 단계가 다음 단계로 자연스럽게 이어져야 한다. 독자가 "다음은 뭐지?"라고 궁금해하며 스크롤을 내리게 만드는 것, 그것이 좋은 TIL 문서의 본질이다.
