# GenOS 패치노트 형식

## 목적

구현 목록이 아니라 고객사 업데이트 검토자가 읽고 기능 변화, 운영 영향, 검증 범위를 이해할 수 있는 릴리스 문서를 만든다. 코드 경로와 commit hash는 조사 근거로만 사용하고 최종 노트에는 싣지 않는다.

## 파일명과 frontmatter

기본 파일명:

```text
GenOS v<version> 패치노트.md
```

기본 frontmatter:

```yaml
---
source:
  - GenOS 플랫폼 v<version> 공식 릴리스 항목
  - v<from> → v<to> 릴리스 비교
related_notes:
  - "[[GenOS 지식 MOC]]"
tags:
  - work/genos
  - work/fde
  - type/reference
  - feature/release-notes
created: YYYY-MM-DD
---
```

실제로 연관된 기존 노트만 `related_notes`에 추가한다. 존재하지 않는 노트 링크를 만들어 채우지 않는다.

## 본문 구조

다음 순서를 기본으로 사용한다.

```markdown
# GenOS v<version> 패치노트

> [!info] 문서 상태
> 어떤 자료를 대조했고 무엇에는 접근하지 못했는지 기록한다.

## 한 줄 결론

## 작성 기준과 범위

## 변경사항 한눈에 보기

## 신규 기능 및 고도화

## 사용성 및 성능 개선

## 버그 수정

## 공개 범위에서 제외된 항목

## 고객사 업데이트 전 확인사항

## 업데이트 후 권장 확인 시나리오

## 헷갈리기 쉬운 지점
```

해당 항목이 전혀 없으면 억지로 내용을 만들지 않는다. 다만 섹션을 생략할 경우 한눈에 보기나 작성 범위에서 해당 분류가 없음을 분명히 한다.

`## 다음에 확인할 것` 섹션은 만들지 않는다. 미확인 한계는 문서 상태와 해당 항목 본문에 즉시 기록한다.

## 섹션별 작성 기준

### 한 줄 결론

이번 버전이 기능 중심, 안정화 중심, 보안 중심 중 무엇인지 한 문단으로 설명한다. 기능명을 모두 나열하지 말고 릴리스의 성격과 고객사 관점의 의미를 요약한다.

### 작성 기준과 범위

다음을 bullet로 명시한다.

- 릴리스 버전과 일자
- 비교 기준
- 포함 범위
- 제외 범위
- 구현 근거를 본문에서 생략했다는 안내

### 변경사항 한눈에 보기

다음 3열 표를 사용한다.

| 구분 | 주요 내용 | 사용자 관점의 의미 |
|---|---|---|

`사용자 관점의 의미`에는 내부 모듈명이 아니라 사용자가 새로 할 수 있는 일이나 달라진 경험을 쓴다.

### 신규 기능 및 고도화

공식 기능 하나당 독립된 `###` 항목으로 작성한다. 각 항목은 가능한 범위에서 다음 다섯 부분을 포함한다.

1. **해결하려는 문제**: 이전의 제약이나 운영상 불편
2. **동작 방식**: 화면과 API를 포함한 실제 흐름
3. **사용자 변화**: 누가 무엇을 새로 할 수 있는지
4. **운영 영향**: 설정, 권한, 관측성, 배포, dependency 영향
5. **범위와 경계**: 자동화되지 않는 부분, 별도 설정이 필요한 부분, 유사 기능과의 차이

필요하면 `주요 기능`, `동작 흐름`, `사용자 관점`, `운영 관점`, `알아둘 점` 같은 소제목이나 표를 사용한다. 모든 항목을 똑같은 틀로 기계적으로 반복할 필요는 없지만, 위 정보가 빠지지 않게 한다.

### 사용성 및 성능 개선

단순히 “UI 개선”, “성능 개선”이라고 쓰지 않는다. 다음 전후 관계를 설명한다.

- 이전에는 사용자가 어떤 순서나 우회 절차를 거쳤는가
- 이번 버전에서 화면, 정렬, 상태 표시, 비교 방식이 어떻게 달라졌는가
- 처리량이나 응답 시간이 개선됐다면 어떤 작업 경로가 대상인가
- 실제 수치가 없으면 정량적 향상을 만들어내지 않는다
- 기능 지원 범위와 입력·모델·데이터 제약은 무엇인가

복합 UI 개선은 numbered flow로, 여러 화면의 변화는 표로 정리한다.

### 버그 수정

가능하면 다음 형태로 작성한다.

```text
- 증상: 사용자가 관찰하던 문제
- 영향: 어떤 작업이 실패하거나 잘못 표시됐는지
- 수정 결과: 이번 버전에서 기대되는 동작
```

원인을 확정할 증거가 없으면 내부 원인을 추측하지 않는다.

### 공개 범위에서 제외된 항목

코드, migration, package에 흔적이 있지만 공식 공개 항목이 아닌 기능을 기록한다. “기능이 있다”가 아니라 “배포 준비 흔적은 확인되지만 공개 기능으로 안내하지 않는다”처럼 경계를 명시한다.

### 고객사 업데이트 전 확인사항

표 형식을 우선한다.

| 확인 영역 | 확인 내용 |
|---|---|

DB migration, 신규 환경변수, 외부 endpoint와 인증, proxy·방화벽·DNS, 기존 데이터 호환성, 배포 구조 변경을 실제 diff에서 확인한 범위만 포함한다.

### 업데이트 후 권장 확인 시나리오

기존 기능 smoke test부터 신규 기능까지 실행 순서대로 번호를 매긴다. 고객사마다 기능 사용 여부가 다르면 조건부 문장으로 쓴다.

### 헷갈리기 쉬운 지점

이름이 비슷한 기능, 자동화 범위에 대한 오해, 설정 없이 동작한다고 오해할 수 있는 지점을 짧은 bullet로 정리한다.

## 문체

- 한국어로 작성하되 제품명, UI label, API·Infra 용어는 원문을 유지한다.
- 고객사와 운영자가 읽는 문서이므로 내부 구현 구조보다 행동과 영향을 먼저 쓴다.
- 확인한 사실은 단정형으로, 해석은 “의미한다”, 미확인은 “확인되지 않았다”로 구분한다.
- 홍보 문구보다 동작 범위와 제약을 정확히 쓴다.
- 같은 내용을 요약 표와 상세 본문에 반복하더라도 상세 본문에는 새로운 설명이 있어야 한다.

## 최종 금지 항목

- commit hash
- 구현 코드 경로와 line number
- 공식 근거 없이 추가한 기능
- 확인하지 않은 성능 수치
- 고객사 커스텀 변경을 플랫폼 기본 기능처럼 표현한 문장
- `다음에 확인할 것` checklist
